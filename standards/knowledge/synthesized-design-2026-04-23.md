# feat: Implement Sub-agent Delegation Protocol + Coverage Evidence Toolchain

## Status

Proposed — awaiting ratification by the AgenticOS standards design team.

---

## Problem Statement

AgenticOS's guardrail chain currently handles project/branch alignment and PR scope, but two critical gaps remain:

**Gap 1 — Delegation without traceability.** When the main agent delegates a review or analysis task to a sub-agent, there is no bounded protocol governing what context the sub-agent receives, what it must produce, or how the main agent validates the result before acting on it. Sub-agents make implicit decisions (e.g., assuming a certain project state) that the main agent cannot see until it is too late. This is the canonical context fragmentation failure.

**Gap 2 — Coverage without evidence.** Test coverage is not configured in the CI pipeline. There is no mechanism to enforce coverage thresholds, to surface per-file coverage as guardrail evidence, or to distinguish "all files above 60%" from "changed files hit 100%" — the latter being the only standard that actually prevents regressions.

Both gaps are exacerbated by the same root cause: **the main agent cannot see what sub-agents did or what the system produced** unless there is a structured artifact and validation layer. The two systems solve different problems, but they share the same architectural pattern: generate evidence → validate evidence → act on it. They belong in the same guardrail chain.

---

## Design Foundation: Cognition Article Principles

This implementation is grounded in the principles from *"Don't Build Multi-Agents"* (Walden Yan, Cognition blog). These are non-negotiable:

1. **Context fragmentation is the #1 failure mode.** Every delegation must carry full, structured context. The sub-agent must never need to guess project state, assumptions, or constraints. The delegation protocol addresses this directly via handoff documents, conversation history, and knowledge map injection.

2. **Decision-making stays centralized.** Sub-agents analyze and report; they never act on their own output. The `validateDelegationOutput()` function is the enforcement gate — the main agent does not read `result.md` until the function passes. This applies equally to coverage: the coverage guardrail blocks or warns, but does not auto-commit or auto-merge.

3. **"Actions carry implicit decisions" — surfaced in Summary Logs.** Every sub-agent action implicitly decides something (e.g., "this file is outdated," "this pattern is a bug"). The `log.md` schema requires explicit `Decisions Made` and `Constraints Respected` fields so the main agent can catch conflicting implicit decisions before they cause problems. This is why the delegation validation checks for these fields by name.

4. **Claude Code model.** Sub-agents answer questions and produce analysis; they do not write code or make binding decisions. The delegation protocol is designed for analysis tasks (review, audit, pattern detection) and does not include provisions for a sub-agent to directly modify source files.

## Integrated Design: Two Protocols, One Guardrail Chain

These two systems are independent in scope but unified in architecture. They slot into the guardrail chain as follows:

```
agenticos_preflight
  ├── Session/project alignment (existing)
  ├── Worktree branch check (existing)
  ├── [NEW] Coverage evidence check    ← guardrail: CoverageValidationResult
  └── Scope validation (existing)

[Main agent starts issue work]
  ├── [sub-agent delegation begins]
  │     → Sub-agent receives handoff.md + context
  │     → Sub-agent produces log.md + result.md
  ├── [NEW] validateDelegationOutput()  ← guardrail: ValidationResult
  │     → If pass: main agent acts on result.md
  │     → If fail: re-delegate with same delegation_id + errors
  ├── agenticos_pr_scope_check (existing)
  └── agenticos_record (with delegation field + coverage evidence)
```

## Part I: Sub-agent Delegation Protocol

### What it does

Defines a bounded, auditable, enforceable protocol for invoking a sub-agent for analysis or review tasks. The protocol covers: context injection, result artifact schemas, quality guardrails, and integration with the existing `agenticos_record` surface.

### Core concepts

**`delegation_id`** (UUID v4) — Generated by the main agent before delegation begins. It is embedded in every file the sub-agent produces and is the primary key for all validation. The sub-agent never generates or reconstructs this ID.

**Handoff document** (`handoff.md`) — Written by the main agent to `standards/.context/delegations/{delegation_id}/handoff.md` before delegation. This is the sub-agent's authoritative instruction and must include: project identity, delegation_id, task scope, required context (conversations, knowledge map, issue file), verification restatement, and return contract.

**Summary log** (`log.md`) — Required output written by the sub-agent before it concludes. It is the execution audit trail and must contain: delegation_id, recorded_at (ISO 8601), sub_task, status (completed/blocked/partial), Actions Taken, Findings, Decisions Made, Constraints Respected, Residual Risks. Missing required sections cause validation to fail.

**Result document** (`result.md`) — Required output written by the sub-agent as the deliverable. Must contain: Summary (3-5 sentences), Findings, Recommendations, Verification Evidence, Delegation ID, Timestamp (ISO 8601).

**`validateDelegationOutput()`** — Pure synchronous function in `mcp-server/src/utils/delegation-validation.ts`. Takes `logPath`, `resultPath`, and `delegationId`. Returns `ValidationResult` with per-field checks. Zero I/O, zero side effects. Called by the main agent before reading `result.md`.

**Re-delegation rule** — If validation fails, the main agent re-invokes the sub-agent with the same `delegation_id` and the validation errors as additional context. The original `handoff.md` remains authoritative. Only the output artifacts are overwritten.

### File layout

```
{projectPath}/standards/.context/delegations/{delegation_id}/
├── handoff.md    # Main agent → sub-agent instruction (written before delegation)
├── log.md        # Sub-agent → main agent execution audit (required output)
└── result.md     # Sub-agent → main agent deliverable (required output)
```

The directory is in `.context/delegations/` — runtime-only, not committed.

### Integration with `agenticos_record`

When `agenticos_record` is called with a `delegation` argument, it:
1. Calls `validateDelegationOutput()` on the delegation's `log_path` and `result_path`
2. Records `validation_pass: boolean` and `validation_errors?: string[]`
3. Appends a delegation summary block to the daily conversation file

### Integration with guardrail evidence

A new `GuardrailCommand` variant `agenticos_validate_delegation` is added, with a corresponding `delegation_validation` slot in `GuardrailEvidenceSlot`. The slot stores the `ValidationResult` from `validateDelegationOutput()`.

### New MCP tool: `agenticos_validate_delegation`

Thin wrapper around `validateDelegationOutput()` — accepts `delegation_id`, optional `log_path`/`result_path`/`project_path`. Returns the full `ValidationResult` as JSON. Passive (no writes).

## Part II: Coverage Evidence Toolchain

### What it does

Adds structured test coverage configuration, per-file evidence generation, CI enforcement on both aggregate thresholds (floor) and changed-scope thresholds (aspirational), and guardrail integration that can block or warn based on context.

### Core concepts

**Aggregate thresholds (CI floor):** Lines 60%, Functions 60%, Branches 50%, Statements 60%. These are enforced in the `build` job. They are honest baselines — not aspirational, not blocking on normal development.

**Changed-scope enforcement (CI gate):** In a PR, all files changed in the diff must have 100% line + function coverage. This is enforced in a dedicated `coverage-changed-scope` job that runs after the build job succeeds. It fails the PR if a changed file drops below 100%. This provides the aspirational goal without blocking development on legacy code.

**Evidence file** (`coverage-evidence.json`): Machine-readable artifact written after each test run. Consumed by the guardrail chain. Contains: version, generatedAt, branch, commit, isPr, prNumber, baseBranch, changedFiles, aggregate percentages, threshold pass/fail, passReason/failReason, and per-file breakdown.

**`generateCoverageEvidence()`**: Reads `coverage-final.json` (V8 raw output), computes aggregate and per-file coverage, applies threshold logic, returns `CoverageEvidence`. Located in `mcp-server/src/utils/coverage-evidence.ts`.

**`validateCoverageEvidence()`**: Reads the evidence file, produces `CoverageValidationResult` with `pass`, `blocking`, and `message`. Called by `agenticos_preflight` and the `agenticos_coverage_check` MCP tool. Located in `mcp-server/src/guardrail/validate-coverage.ts`.

**Two enforcement modes**: `strict` (block if not passed — used on PR push) and `advisory` (warn but do not block — used on local preflight). The mode is a parameter passed to the guardrail call.

**UPDATE_BASELINE mechanism**: Set the `COVERAGE_UPDATE_BASELINE` repository variable to `1` to skip threshold enforcement for a specific run. Used when intentionally adding a new file without tests, or when the gap is acknowledged and tracked.

### Configuration surface

| Variable | Default | Purpose |
|----------|---------|---------|
| `COVERAGE_UPDATE_BASELINE` | `0` | Skip threshold enforcement for one run |
| `COVERAGE_THRESHOLD_LINES` | `60` | Adjustable without code change |
| `COVERAGE_THRESHOLD_FUNCTIONS` | `60` | Adjustable without code change |
| `COVERAGE_THRESHOLD_BRANCHES` | `50` | Adjustable without code change |
| `COVERAGE_THRESHOLD_STATEMENTS` | `60` | Adjustable without code change |

### Integration with guardrail evidence

Coverage validation is the **final step before scope validation** in `agenticos_preflight`. The evidence file is written to `mcp-server/coverage/coverage-evidence.json` and consumed by the guardrail call in `agenticos_preflight`.

## Implementation Phases

### Phase 1: Foundation — Delegation Protocol Core

**Owner:** Implementation team (any agent can take this)

**Goal:** Establish the delegation protocol as the standard mechanism for sub-agent tasks.

1. Create `mcp-server/src/utils/delegation-validation.ts` with `validateDelegationOutput()` — pure function, no I/O, zero side effects
2. Update `tasks/templates/sub-agent-handoff.md` to include `delegation_id` field, `Return Contract`, required context fields, and verification restatement
3. Mirror updated template to `projects/agenticos/.meta/templates/sub-agent-handoff.md`
4. Create `mcp-server/src/tools/validate-delegation.ts` as thin MCP wrapper around the pure function
5. Register `validate-delegation` in `mcp-server/src/tools/index.ts`
6. Extend `mcp-server/src/tools/record.ts` with `delegation` field handling, validation call, and conversation append
7. Extend `mcp-server/src/utils/guardrail-evidence.ts` with `agenticos_validate_delegation` command and `delegation_validation` slot

**Exit criteria:** A main agent can generate a `delegation_id`, write a handoff, invoke a sub-agent, call `validateDelegationOutput()`, and record the result via `agenticos_record`. All required schema fields are enforced.

**Estimated complexity:** Medium. The core logic is a single pure function; integration is extension of existing surfaces.

---

### Phase 2: Foundation — Coverage Toolchain Core

**Owner:** Implementation team

**Goal:** Get coverage running, producing evidence, and visible in CI.

1. Add `coverage` block to `mcp-server/vitest.config.ts` with v8 provider, 60/60/50/60 thresholds, and per-module overrides for `guardrail/` (80%) and `mcp/handlers/` (70%)
2. Add `test:coverage` script to `mcp-server/package.json`
3. Add coverage upload step to `.github/workflows/ci.yml` `build` job (upload `mcp-server/coverage/` as artifact)
4. Create `tools/coverage-preflight.sh` as bash guardrail script
5. Create `mcp-server/src/utils/coverage-evidence.ts` with `generateCoverageEvidence()` and types
6. Create `mcp-server/src/guardrail/validate-coverage.ts` with `validateCoverageEvidence()`
7. Create `mcp-server/src/tools/coverage-check.ts` as `agenticos_coverage_check` MCP tool
8. Add `coverage-preflight.sh` call to `agenticos_preflight` as the final pre-scope step

**Exit criteria:** `npm run test:coverage` runs in CI with thresholds enforced. `coverage-evidence.json` is produced and written to `mcp-server/coverage/`. `agenticos_coverage_check` tool returns structured `CoverageValidationResult`.

**Estimated complexity:** Medium. The Vitest configuration is well-documented. The evidence generator is straightforward. CI integration uses standard artifact upload/download.

---

### Phase 3: Changed-Scope Enforcement

**Owner:** Implementation team

**Goal:** Enforce 100% coverage on changed files in PRs.

1. Add `coverage-changed-scope` job to `.github/workflows/ci.yml` — runs after `build`, only on PRs, downloads coverage artifact, computes changed files vs base branch, fails if any changed source file is below 100% line/function coverage
2. Add `UPDATE_BASELINE` handling in `coverage-changed-scope` job
3. Add PR comment automation for coverage gaps (uses `actions/github-script` to post a coverage gap comment on the PR)

**Exit criteria:** A PR that changes a source file without adding corresponding tests fails the CI changed-scope job. The `COVERAGE_UPDATE_BASELINE` variable correctly bypasses the enforcement for acknowledged gaps.

**Estimated complexity:** Low-medium. The changed-scope script is already designed (see Section 2.2 of coverage-toolchain-design.md). CI job orchestration is standard.

---

### Phase 4: Delegation Guardrail Integration

**Owner:** Implementation team

**Goal:** Make delegation validation a first-class guardrail command in the chain.

1. Wire `agenticos_validate_delegation` into the guardrail chain — fires between sub-agent return and main agent reading `result.md`
2. Document re-delegation rule: same `delegation_id`, validation errors as additional context, original `handoff.md` remains authoritative
3. Archive ratified delegation protocol to `standards/knowledge/delegation-protocol-YYYY-MM-DD.md`
4. Add end-to-end integration test: main agent delegates, sub-agent produces artifacts, guardrail validates, `agenticos_record` records

**Exit criteria:** The delegation guardrail fires automatically when a delegation is recorded. Failed validations surface actionable errors (not just "validation failed"). Re-delegation with the same `delegation_id` correctly overwrites only the output artifacts.

**Estimated complexity:** Low. The guardrail integration is purely wiring — the validation function and tool already exist from Phase 1.

---

### Phase 5: Knowledge Archival and Rollout

**Owner:** Standards design team

**Goal:** Ensure both protocols are documented, discoverable, and adopted as standard practice.

1. Archive `coverage-toolchain-design.md` as ratified knowledge to `standards/knowledge/coverage-toolchain-YYYY-MM-DD.md`
2. Archive delegation protocol (from `delegation-protocol-design.md`) as ratified knowledge
3. Update `standards/knowledge/index.yaml` to include both entries with appropriate tags
4. Add both protocols to the agent onboarding materials
5. Review all existing delegation calls in the codebase and migrate to the new protocol

**Exit criteria:** Both protocols appear in `knowledge/index.yaml`. New agents reading the knowledge index can discover and adopt the protocols without verbal handoff.

**Estimated complexity:** Low. Primarily documentation and knowledge management.

## Priority Ordering

```
P0 (must do in this repo, unblocks guardrail chain):
  Phase 1 (Delegation Protocol Core) — enables structured sub-agent work
  Phase 2 (Coverage Toolchain Core) — enables coverage evidence generation

P1 (follow-up, improves quality gates):
  Phase 3 (Changed-Scope Enforcement) — makes coverage a real PR gate
  Phase 4 (Delegation Guardrail Integration) — makes delegation validation automatic

P2 (rollout, adoption):
  Phase 5 (Knowledge Archival and Rollout) — ensures discoverability and adoption
```

**Rationale:** Phases 1 and 2 are foundational — they establish the evidence generation and validation patterns that the guardrail chain depends on. Phases 3 and 4 build on top of them to create real enforcement. Phase 5 is the adoption layer.

Both Phase 1 and Phase 2 can be implemented in parallel by different agents. The coverage toolchain does not depend on the delegation protocol (and vice versa). Phase 3 depends on Phase 2. Phase 4 depends on Phase 1. Phase 5 depends on both.

## Summary: What Gets Built and Why

| Phase | What | Why it matters |
|-------|------|---------------|
| Phase 1 | `validateDelegationOutput()`, handoff template, MCP tool, record extension | Sub-agents produce trustworthy, auditable output. Main agent never acts on unvalidated results. |
| Phase 2 | Coverage config, evidence generator, guardrail function, MCP tool, preflight script | Coverage runs in CI and produces machine-readable evidence. Guardrail can read and act on it. |
| Phase 3 | Changed-scope CI job, UPDATE_BASELINE, PR comment automation | PRs that degrade coverage on changed files fail automatically. No manual review needed. |
| Phase 4 | Delegation as first-class guardrail command | Delegation validation fires automatically. Failed validations surface actionable errors. |
| Phase 5 | Knowledge archival | Both protocols are discoverable, adoptable, and self-documenting via the knowledge index. |
