## OKR编写风格指南（基于2025年全年及Q1-Q4分析）

### 零、OKR标准写法核心原则

#### 0.1 OKR基本定义
- **Objective (O)**：目标，定性描述"要去哪里"，鼓舞人心、有挑战性
- **Key Result (KR)**：关键结果，量化回答"如何知道到了那里"，可衡量、有截止时间

#### 0.2 Objective 标准要求
| 要求 | 说明 | 2025年实践示例 |
|------|------|---------------|
| **定性为主** | O是方向性描述，不是具体数字 | `降本增效`、`交付稳定性`、`AI创新应用` |
| **鼓舞人心** | 读起来有动力，指向有意义的结果 | `使得技术成为其中一个核心竞争力` |
| **有挑战性** | 需要努力才能达成，非轻易可得 | `全面建成xxx`、`实现xxx新范式` |
| **聚焦关键** | 每季度/年度3-4个O，不贪多 | 年度3个O，季度3-4个O |
| **对齐战略** | 承接公司/部门战略方向 | `为公司「AI+金融」大战略奠定基础` |

#### 0.3 Key Result 标准要求
| 要求 | 说明 | 正确示例 | 错误示例 |
|------|------|---------|---------|
| **可量化** | 必须有明确数字或可验证标准 | `SLA达成率>90%` | `提升SLA达成率` |
| **结果导向** | 描述结果状态，非过程任务 | `人力投入减少>30%` | `优化人力分配流程` |
| **有挑战性** | 全部达成说明目标不够挑战 | 建议达成率70-80% | 100%轻松达成 |
| **3-5个KR** | 每个O下KR数量适中 | 2-4个KR | 超过5个KR |
| **相互独立** | KR之间无依赖，可并行推进 | 各KR独立衡量 | KR2依赖KR1完成 |

#### 0.4 常见错误及避免

**❌ 把任务当KR（Task vs Result）**
```
错误：KR1: 完成测试环境治理项目
正确：KR1: 通过测试环境治理，使得银数环境成本相比1月下降20%
```

**❌ KR无法衡量**
```
错误：KR2: 提升团队协作效率
正确：KR2: 通过建立虚拟小组机制，使得虚拟小组成员能够独立解决机构80%生产问题
```

**❌ O过于具体（写成了KR）**
```
错误：[O1]测试环境成本下降20%
正确：[O1]降本增效：xxx，研发降本增效明显
```

**❌ O过于抽象（缺乏方向感）**
```
错误：[O1]做得更好
正确：[O1]降本增效：新机构交付周期缩短重大突破，交付人力成本及测试环境资源成本「双降」
```

#### 0.5 OKR层级关系
```
公司战略
   ↓ 承接
部门年度OKR（本文档）
   ↓ 分解
季度OKR（Q1→Q2→Q3→Q4递进）
   ↓ 落地
个人/小组OKR（可选）
```

#### 0.6 OKR评分标准（参考）
| 得分 | 含义 | 建议 |
|------|------|------|
| 0.0-0.3 | 未能取得进展 | 分析原因，调整策略 |
| 0.4-0.6 | 取得进展但未达预期 | 可接受，继续努力 |
| 0.7-0.8 | 达成挑战目标 | 理想状态 |
| 0.9-1.0 | 完全达成或超额 | 可能目标不够挑战 |

### 一、整体结构规范

#### 1.1 Objective (O) 格式
```
[Ox]主题词：描述句。
```

**规则：**
- 使用 `[O1]`、`[O2]`、`[O3]`、`[O4]` 标记，数字连续
- 主题词：2-4个字的名词短语，简洁有力
- 冒号使用**英文冒号**`:`后无空格，或**中文冒号**`：`
- 描述句：完整句子，强调最终状态/价值

**主题词历史沿革：**

| 年份/期间 | 常用主题词 |
|----------|-----------|
| 2024年度（大范围） | 金科交付、数据基建、技术基建、技术服务、组织建设 |
| 2024Q2 | 金科交付、组织建设 |
| 2024Q3-Q4 | 研发提效、交付稳定性、团队建设 |
| 2025年度 | 降本增效、交付稳定性、团队建设 |
| 2025Q1 | 降本增效、交付稳定性、团队建设 |
| 2025Q2 | AI创新应用（O0过渡）、降本增效、交付稳定性、团队建设 |
| 2025Q3-Q4 | AI创新应用（O1）、降本增效、交付稳定性、团队建设 |

**注意**：2024年度OKR覆盖多个业务线（���科+数据+基础技术），O 数量达 5 个；2025年起聚焦金科研发部，年度保持 3 个O。

**O编号特殊情况：**
- 2025Q2 曾短暂使用 `[O0]` 作为 AI 创新应用的编号（AI 主题从 Q2 开始纳入，Q3 正式升为 O1）
- 正常编号从 `[O1]` 开始，不存在 `[O0]`

**描述句常用句式（实证整理）：**
- `xxx，体现技术在业务实现、降本提效等方面的重要价值`
- `xxx，xxx，体现技术在xxx方面的杠杆作用`
- `xxx，为xxx奠定基础`
- `xxx，研发降本增效明显`
- `xxx，xxx明显提升`
- ��个动宾短语并列，用逗号分隔，如：`全面建成xxx，xxx落地实践，大幅优化xxx，体现xxx`
- 结尾常用：`使得xxx成为其中一个xxx`（注意：用"其中一个"而非"之一"）
- 简洁陈述式（多见于交付稳定性/团队建设O）：`生产服务稳定可靠，保持高效交付状态，为机构业务开展提供稳定高效的基础`
- 状态达成式（团队建设O）：`大家有清晰的责任感、使命感，以业务成功为目标驱动各项工作`

#### 1.2 Key Result (KR) 格式
```
KRx:[标签]通过xxx，使得：
	a. 具体指标1。
	b. 具体指标2。
```

**规则：**
- 使用 `KR1:`、`KR2:`、`KR3:`、`KR4:` 标记
- 标签格式：`[标签]`（方括号，2025年主流）或 `「标签」`（书名号前置，2024年常见）
- **核心句式**：`通过xxx，使得：`（这是最重要的风格特征）
- 子项使用 `a.` `b.` `c.` 或 `1.` `2.` `3.` 编号
- **子项编号演进规律**：2024年全用字母（a.b.c.），2025Q2起混用，2025Q3-Q4 以数字（1.2.3.）为主；年度OKR 习惯用字母
- 子项前使用Tab缩进
- **季度OKR** 的 KR 后可附 `@人名` 标注负责人，年度OKR无此用法

**子项特殊格式：**
- 场景分类��：`场景名：具体描述`，如`代码开发场景：开发质量有质的提升`
- 指标分类式：`指标类型：具体数值`，如`人力模型优化：新机构接入人力投入减少>30%`
- 子领域标签式（2024年常见）：`a.「领域」描述`，如`a.「测试」测试效率提升...` `b.「运维」生产运维操作系统化率>95%`
- 多维度分隔：使用分号`；`分隔不同维度，如`开发质量有质的提升，单版本缺陷密度<1%；交付效率有质的提升`
- 补充说明式：用括号补充，如`P1及以上（1个月内上线）`、`（做的好的，需要改进的）`

**常用 [ ] 标签（2025年主流）：**
- 流程类：`[全流程]`、`[开发]`、`[测试]`、`[运维]`、`[开发测试]`
- 团队类：`[总结分享]`、`[实践��享]`、`[调结构]`、`[招聘]`、`[招聘培训]`、`[团队成长]`、`[新技术]`
- 其他：`[业务赋能]`、`[稳定性]`、`[交付效率]`、`[技术创新提效]`

**常用 「 」 标签（2024年常见，作KR主题前置）：**
- `「质量左移」`、`「流程顺畅」`、`「提效」`、`「配置化」`、`「稳定性」`、`「技术创新提效」`

### 二、量化指标表达规范

#### 2.1 百分比表达
- 大于：`>95%`、`>90%`、`>80%`、`>30%`、`>20%`
- 等于：`100%`、`60%`
- 小于：`<10%`、`<1%`
- **注意**：符号与数字之间无空格

#### 2.2 数量表达
- 故障类：`0严重故障`、`1个P0级别故障`、`0个P0级别故障`
- 机构类：`x家机构`、`3家新机构`、`1-2家机构`
- 人力类：`15人月`、`x人月`、`2人天`
- 场景类：`2个以上场景`、`1个领域`

#### 2.3 时间表达
- 分钟级：`<10分钟`、`<30分钟`
- 天级：`0.5天`、`1天`、`2天`、`1个工作日`
- 周/月级：`1个月内`、`2周内`、`四周内`
- 人天/人月：`5人天`、`15人月`

#### 2.4 比较/变化表达
- 下降类：`下降30%`、`下降>30%`、`缩短75%`、`减少>30%`
- 提升类：`提升>200%`、`效率提升80%`、`提升10%`
- 从...到...：`从5天缩短到4天`、`从2天缩短到0.5天`、`从15人月下降到x人月`
- 相比：`相比1月下降30%`

#### 2.5 级别表达
- 问题级别：`P0`、`P1`、`P2`、`P3`（数字越小越严重）
- 需求级别：`P00`、`P0`、`P1及以上`
- 代码层级：`L1`、`L2`、`L3`
- 范围表达：`P1及以上`、`P3及以上`、`L1及以上`

#### 2.6 待定数值
- 使用小写 `x` 作为占位符：`x家机构`、`x人月`

### 四、句式与语言风格

#### 4.1 核心句式模板

**KR主干句式：**
```
通过xxx建设/优化/升级/深化/完善/梳理，使得：
通过xxx措施/方法/机制，使得：
通过对xxx进一步xxx，使得：
通过聚焦xxx，使得：
通过识别并支持xxx，使得：
通过构建xxx，使得：
```

**措施描述：**
- 加强xxx、强化xxx
- 建立xxx机制/规范/标准/制度
- 引入xxx工具/技术/能力
- 不断完善/持续优化
- 进一步深化/进一步平台化

**结果描述：**
- 使得xxx
- 达成xxx
- 实现xxx
- 形成xxx
- 建立xxx
- 初步实现xxx
- 全面实现xxx

#### 4.2 程度副词使用
- **起步阶段**：初步、初级、探索、尝试
- **进行阶段**：进一步、持续、不断
- **成熟阶段**：全面、完全、高度、大幅
- **质变表达**：有质的提升、明显提升、显著提升

#### 4.3 条件限定表达
- `（除机构原因）`
- `（需求澄清后）`
- `（在研发交付周期内）`
- `（私有化部署）`
- `（除AI项目新增）`

#### 4.4 范围/领域表达
- 局部上、整体上、全面
- 部分场景、部分工作、部分问题
- 各领域、全领域、跨领域
- 端到端（本地到机构）、端到端（需求到测试）

#### 4.5 用词偏好（重要）
| 偏好用词 | 避免用词 | 示例 |
|---------|---------|------|
| 及 | 和 | `业务及产品团队` |
| 能够 | 可以 | `能够对xxx提出` |
| 达成 | 达到（目标语境） | `目标达成` |
| 并 | 且、而且 | `提出xxx并被接受` |
| 其中一个 | 之一 | `成为其中一个核心竞争力` |
| 有质的提升 | 显著提升（避免AI味） | `开发质量有质的提升` |

#### 4.6 特殊KR格式（无"通过...使得"句式）
部分KR可直接描述能力/结果，不使用标准句式：
- `[业务赋能]能够对业务及产品团队提出有价值的建议并被接受落地，以金科业务目标达成为技术团队目标。`
- 常见于团队建设类O的最后一个KR

### 六、格式细节规范

#### 6.1 标点符号
- O标题后的冒号：英文 `:` 或中文 `：`均可，保持一致
- KR后的冒号：英文 `:`
- 子项结尾：句号 `。` 或无标点均可
- 列表分隔：分号 `；` 或逗号 `，`
- **括号使用**：条件限定用全角括号`（）`，如`（除机构原因）`；数字相关可用半角`()`，如`>10家机构(私有化部署)`

#### 6.2 缩进与换行
- KR主干与子项之间换行
- 子项前使用Tab缩进（或空格缩进对齐）
- 每个O之间保留空行
- 文档末尾可保留1-2个空行

#### 6.3 编号规范

### 八、写作检查清单

编写OKR时，请对照以下清单检查：

**一、OKR标准原则检查（方法论层面）：**

- [ ] O是定性描述，鼓舞人心，有挑战性

- [ ] O使用 `[Ox]主题词：描述句` 格式

- [ ] 季度目标符合演进逻辑（Q1探索→Q4成熟）

## 使用说明

### 核心原则：学习风格，而非照搬格式

本文档记录的是从历史 OKR 中提炼出的**写作原则和规律**，而不是"格式模板"。
使用时应理解每条规范背后的逻辑，而不是机械复制历史写法。

**两个层次的区别：**

| 层次 | 内容 | 目的 |
|------|------|------|
| 原则层（为什么） | 为什么 KR 用「通过...使得：」？为什么 O 不含数字？ | 确保 OKR 思维正确 |
| 格式层（怎么写） | 具体的句式、标签、符号格式 | 确保风格统一 |

**原则层核心理解：**
- `通过xxx，使得：` 这个句式强迫把**措施**和**结果**分开，防止把任务当 KR
- O 不含数字，是因为 O 描述**方向**，KR 描述**里程碑**，混用会让团队只关注数字而忘记方向
- KR 要有挑战性（达成率 70-80%），是因为 OKR 不是 KPI，不是考核工具，是拉伸目标

---

### 典型使用场景

**场景一：非结构化素材 → 规范OKR（最常见）**

用户通常提供语音输入转录文本、工作记录、聊天记录等非结构化素材。核心任务是**意图理解，而非格式翻译**。

处理步骤：
1. **解析意图**：提炼方向意图（想去哪）、措施描述（怎么做）、隐含成功标准（算达成的条件）
2. **识别量化缺口**：哪些成功标准还没有具体数值，用 x 占位或主动询问
3. **映射 OKR 结构**：方向意图 → O；措施描述 → KR 主干；成功标准 → KR 子项
4. **风格对齐**：用词、句式、格式与历史 OKR 保持一致（参考本文档规范）

> 示例输入（非结构化）：
> "明年上半年主要是想把新机构交付速度再提升，现在一个机构要两个月，争取降到一个月以内。另外AI这块要继续��，特别是研发助手，希望能让测试自动化比例再高一点。"
>
> 解析：
> - 方向意图：交付效率提升 + AI 在研发流程中的渗透
> - 措施：新机构交付流程优化；AI 研发助手建设
> - 成功标准：交付周期 <1 个月（已量化）；测试自动化比例提升（缺具体目标值 → 用 x 占位）

**场景二：迭代优化已有版本**

在已有OKR基础上，根据反馈或新信息进行修改：
- 指标数值调整
- KR措辞优化
- 增减O或KR
- 风格统一性修正
