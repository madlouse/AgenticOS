# T5T 收集与论证规则

> 本规则定义 Step 1（收集与论证材料）的执行标准
>
> 核心原则：LLM 语义理解，而非关键词匹配；碎片化对话也是工作内容。

---

## 1. 收集规则

### 1.0 前提：代入角色，获取年度目标

**开始收集前，必须先读取 `role-and-okr.md`，明确：**
- 当前角色：金科研发部负责人
- 年度目标体系：AI产品交付、新机构交付、质量加固、效率提升等主线
- 我的战略重点是什么？

### 1.1 获取群组成员

```text
优先从缓存读取：~/AgenticOS/projects/t5t/cache/team-members.json
缓存不存在或需要刷新时：
  命令：opencli 360teams read --target Nn3vpbCjTz8sttlK8I2QkM --type GROUP --limit 50
  识别成员（排除自己），提取姓名和员工ID
  保存到缓存文件
```

### 1.2 获取下属私聊消息

```text
命令：opencli 360teams read --target {员工ID} --type PRIVATE --limit 30
时间范围：最近 3 天
```

### 1.3 内容理解策略（核心）

**必须使用 LLM 语义理解，不能只做关键词匹配**

```text
将成员的所有消息作为整体上下文，使用 LLM 语义理解分析：
  - 消息讨论了哪些项目/工作
  - 本周做了什么、进展如何
  - 下周计划做什么

特别注意：
  - 碎片化对话（如“三峡初步的”、“下周一就会进入”）也是工作内容
  - 需要语义推断来理解完整含义
  - 如果只有碎片化消息，标注“需语义推断”
```

### 1.4 保存原始素材

```text
路径：~/AgenticOS/projects/t5t/Week-YYYY-MM-N/raw-materials/
文件命名：成员名-日期.md
内容：LLM 语义理解后的完整周报内容
```

### 1.5 角色代入判断

**在理解每条素材时，用负责人视角判断：**

```text
- 这件事对部门意味着什么？
- 它属于哪条战略主线？（参考 role-and-okr.md）
- 它是延续性事项还是新事项？
- 需要我下周关注或推动什么？
```

## 2. 论证规则

### 2.1 论证检查项

| 检查项 | 标准 |
|--------|------|
| 完整性 | 每个成员是否有周报？缺失的是否有原因说明？ |
| 事实性 | 内容是否有原始消息支撑？是否有捏造？ |
| 清晰性 | 下周计划是否明确？模糊的需标注 |

### 2.2 论证输出格式

```markdown
## 素材论证结果

| 成员 | 论证结果 | 备注 |
|------|----------|------|
| 雷柴卫 | ✓ 30条私聊，语义理解完整 | 三峡项目需后续跟进确认 |
| 黄俊林 | ✓ 结构化周报，内容清晰 | - |
```

## 3. 事实汇总规则

### 3.1 汇总原则

**只堆砌原始事实，不加工**

```text
- 不做语义升华
- 不合并同类项
- 不删除“不重要”的信息（由用户判断）
- 严格按维度分类
```

### 3.2 维度体系（参考 `t5-writing-rules.md`）

| 维度 | 典型内容 |
|------|----------|
| 新机构交付项目 | 石嘴山、兴业、三峡、汉口、兰州等 |
| AI信贷员项目 | 常熟银行AI信贷员版本迭代 |
| 全流程质量加固 | 监控指标、兜底方案、错误码 |
| 效率效能提升 | Agentic DevOps/Test/Dev、自动化测试 |
| 基础设施降本 | DB/GPU资源回收 |
| 发现能力建设 | 差异化监控、日志编码、用户体感 |
| 团队管理 | 方法演示、验收 |

### 3.3 汇总格式

```markdown
## 事实材料汇总（Week N）

### 上周T5T计划回顾
| # | 事项 | 状态 |
|---|------|------|
| 1 | ... | 🔄进行中 |

### 维度X：XXX
| 机构/事项 | 负责人 | 本周进展 | 下周计划 |
|------------|--------|----------|----------|
| ... | ... | ... | ... |
```

### 3.4 状态标注

| 标注 | 含义 |
|------|------|
| ✅ | 已完成 |
| 🔄 | 进行中 |
| ⚠️ | 有风险/需跟进 |
| ❓ | 待确认 |

## 4. 用户确认要求

**只有用户确认后，才能进入 Step 2**

确认清单：
- [ ] 事实材料是否完整？
- [ ] 是否有遗漏或偏差？
- [ ] 维度划分是否合理？
- [ ] 是否可以进入编写阶段？
